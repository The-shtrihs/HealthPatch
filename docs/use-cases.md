# Health Patch — Use Cases

> Детальний опис юзкейсів системи Health Patch за кожним доменом.

## 1. Identity & Auth

### UC-01 — Реєстрація користувача

| Поле | Значення |
|------|----------|
| **Назва** | Реєстрація користувача |
| **Актор** | Гість (незареєстрований користувач) |
| **Передумови** | Користувач не має облікового запису в системі |
| **Основний сценарій** | 1. Користувач надсилає `POST /auth/register` з полями: `name`, `email`, `password`, `password_confirm`<br>2. Система перевіряє унікальність email<br>3. Система валідує пароль (мін. 8 символів, великі та малі літери, цифра, спеціальний символ `@$!%*?&_`)<br>4. Система хешує пароль алгоритмом Argon2<br>5. Створюється запис USER з `is_verified=false`<br>6. Надсилається лист для верифікації email (фонова задача)<br>7. Повертається `201 Created` з повідомленням про успіх |
| **Альтернативні сценарії / помилки** | — Email вже існує → `409 Conflict` (EmailAlreadyExistsError)<br>— Паролі не збігаються → `422 Unprocessable Entity`<br>— Пароль не відповідає вимогам складності → `422 Unprocessable Entity` |

### UC-02 — Авторизація (логін)

| Поле | Значення |
|------|----------|
| **Назва** | Авторизація користувача |
| **Актор** | Зареєстрований користувач |
| **Передумови** | Обліковий запис існує |
| **Основний сценарій** | 1. Користувач надсилає `POST /auth/login` з `email` та `password`<br>2. Система перевіряє хеш Argon2<br>3. Якщо 2FA вимкнено: генерується access token (1 год) та refresh token (7 днів)<br>4. Повертається `200 OK` з `access_token`, `refresh_token`, `token_type`, `expires_in` |
| **Альтернативні сценарії / помилки** | — Невірні облікові дані → `401 Unauthorized` (InvalidCredentialsError)<br>— Обліковий запис деактивовано (`is_active=false`) → `401 Unauthorized`<br>— 2FA увімкнено → повертається тимчасовий `temp_token` з типом `2fa_required`; потрібна верифікація через `POST /auth/verify-2fa` |

### UC-03 — OAuth авторизація

| Поле | Значення |
|------|----------|
| **Назва** | Авторизація через зовнішній провайдер |
| **Актор** | Гість або зареєстрований користувач |
| **Передумови** | OAuth провайдер налаштований (Google, GitHub або Facebook) |
| **Основний сценарій** | 1. Користувач переходить на `GET /oauth/{provider}` (де provider — google, github або facebook)<br>2. Система генерує OAuth state (зберігається в Redis з TTL 5 хв) та перенаправляє на провайдера<br>3. Користувач автентифікується у провайдера<br>4. Провайдер перенаправляє на `GET /oauth/{provider}/callback` з `code` та `state`<br>5. Система перевіряє state, обмінює code на інформацію про користувача<br>6. Якщо новий користувач: створюється USER з `oauth_provider`, `password_hash=NULL`, `is_verified=true`<br>7. Генеруються токени та повертаються через redirect |
| **Альтернативні сценарії / помилки** | — Невалідний state → `400 Bad Request` (OAuthStateMismatchError)<br>— Провайдер повернув помилку → відповідна помилка<br>— Email вже зайнятий іншим обліковим записом → `409 Conflict` |

### UC-11 — Увімкнення двофакторної автентифікації

| Поле | Значення |
|------|----------|
| **Назва** | Увімкнення 2FA (TOTP) |
| **Актор** | Автентифікований користувач |
| **Передумови** | Користувач автентифікований; 2FA ще не увімкнено |
| **Основний сценарій** | 1. Користувач надсилає `POST /auth/enable-2fa`<br>2. Система генерує TOTP-секрет та QR-код (base64)<br>3. Повертається `200 OK` з `secret`, `qr_code_base64`, `message`<br>4. Користувач сканує QR-код додатком автентифікатора<br>5. Користувач надсилає `POST /auth/confirm-2fa?code=...` з 6-значним кодом<br>6. Система перевіряє код, встановлює `is_2fa_enabled=true`<br>7. Повертається `200 OK` |
| **Альтернативні сценарії / помилки** | — Невірний TOTP-код при підтвердженні → помилка<br>— 2FA вже увімкнено → помилка |
