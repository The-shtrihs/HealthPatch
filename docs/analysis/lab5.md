## 1. Аналіз Lab 5

### 1.1. Вибір меж контекстів і модулів

Систему поділено на **два макроконтексти** (Bounded Contexts):

| Макроконтекст | Призначення | Шлях у репозиторії |
|---|---|---|
| **Core** | Активна бізнес-логіка: ідентичність, активності, харчування, гейміфікація, профіль | `src/core_context/` |
| **Analytics** | Read-only споживач: аудит-журнал та read-models (проекції) для звітності | `src/analytics_context/` |

Усередині **Core** виділено п’ять незалежних модулів (mini-bounded contexts), кожен з повним DDD-стеком (`domain / application / infrastructure / presentation`):

- `auth` — реєстрація, логін, токени, OAuth, 2FA;
- `user` — профіль користувача, фітнес-дані;
- `activity` — тренування, плани, персональні рекорди;
- `nutrition` — щоденник харчування, добові норми;
- `gamification` — XP, рівні, нагороди.

**Принцип розрізу.** Межі модулів обиралися за **інваріантами агрегатів** і **транзакційними кордонами**: дані, що мають змінюватись разом у межах одного запиту з гарантією узгодженості, належать до одного модуля. Наприклад, `MealEntry` і `DailyDiary.totals` — один транзакційний інваріант (Nutrition); нарахування XP за ту ж саму подію — окремий інваріант (Gamification), бо за бізнес-логікою воно має право бути асинхронним і навіть може запізнитись.

**Чому Analytics — окремий макроконтекст.** Аудит та read-моделі є *вторинним* представленням стану системи, потрібним для звітності та compliance. Виділення в окремий контекст:
- знімає з Core навантаження агрегації (write-path лишається швидким);
- дозволяє Analytics еволюціонувати незалежно (нові дашборди, нові проекції без ризику для бізнес-write-path);
- природно фіксує напрям залежності: **Analytics залежить від Core, але не навпаки** (це примусово перевіряється `import-linter`).

### 1.2. Як ACL захищає модулі

Кожен модуль експонує лише пакет `contracts/`:
- `contracts/dtos.py` — публічні DTO (Pydantic v2, frozen);
- `contracts/events.py` — інтеграційні події у минулому часі (`WorkoutCompleted`, `MealEntryAdded`, …);
- `contracts/ports.py` — інтерфейси-порти (`IUserDirectory`, `IExerciseCatalog`, …);
- `contracts/dependencies.py` — публічні FastAPI dependencies (наприклад, `get_current_user` для Auth).

Усі інші шари модуля (`domain`, `application`, `infrastructure`, `presentation`) — **приватні**. Заборону на пряме «прокидання» в чужі приватні шари формалізовано декларативно у `pyproject.toml` через `import-linter` (контракт `Core modules expose only contracts/`). Будь-яка спроба `from src.core_context.user.domain.models import …` зі сторони Nutrition завалить лінт у CI.

**ACL (Anti-Corruption Layer)** живе у пакетах `acl/` модуля-споживача. Приклад — `src/core_context/gamification/acl/translators.py`. Замість того, щоб приймати `WorkoutCompletedEvent` з `activity.domain` або власне `MealEntryAdded` із nutrition.contracts напряму, Gamification:

1. підписується на **інтеграційну подію** з `activity.contracts.events.WorkoutCompleted` (мінімальний контракт: `user_id`, `duration_minutes`, `total_volume_kg`);
2. передає її через `from_workout_completed(event) -> XpAwardCommand` — це і є межа ACL;
3. далі вся внутрішня логіка оперує лише `XpAwardCommand` — рідною мовою модуля Gamification.

Ще один зразок ACL — Analytics→Core: `src/analytics_context/audit/acl/translators.py`. Будь-яка інтеграційна подія з `auth.contracts.events`, `activity.contracts.events`, `nutrition.contracts.events` перетворюється на внутрішній `AuditEntry`. За межі цього файлу типи Core не виходять.

Як Analytics-модуль використовує перекладений тип `UserProfileDTO`, який надав User-модуль, демонструє інший приклад: Nutrition отримує `FitnessGoal`/`Gender` не з `user.domain`, а з `user.contracts.dtos` — той самий `StrEnum`, але оголошений у публічному контракті, і відтак сама присутність ACL-перекладу не потрібна (контракт спеціально спроєктовано як вже придатний до прямого вживання).

### 1.3. Strong vs Eventual Consistency — обґрунтування поділу

| Гарантія | Де застосовано | Реалізація |
|---|---|---|
| **Strong consistency** | Усередині одного модуля, в межах одного UoW: `MealEntry` + `DailyDiary.totals` + `MealEntryAddedEvent` (внутрішній) комітяться однією PostgreSQL-транзакцією. | `src/shared/infrastructure/base_uow.py` (savepoint-aware async UoW) + `dispatch_domain_events` після успішного `commit`. |
| **Eventual consistency** | Між модулями: реакції Gamification на `WorkoutCompleted`; запис аудиту в Analytics; оновлення проекцій. | Подія публікується у `EventBus` після коміту модуля-джерела. Споживачі або виконуються синхронно (`mode="sync"` — для UX-критичних, як XP), або асинхронно через arq (`mode="async"` — для аудиту, проекцій, e-mail). |

**Аргументація.** Розтягувати транзакції на кілька модулів через 2PC або XA — це руйнує саму ідею модульності: відмова одного модуля блокує всю систему, схеми переплутуються, тестування стає кошмаром. Натомість:

- **Усередині модуля** транзакція дешева, інваріанти прості, користувач має право очікувати «все або нічого». Strong consistency тут безкоштовна.
- **Між модулями** допустимо, що нарахування XP, запис в аудит-лог чи оновлення лідерборду «доганяють» через секунду. Eventual consistency натомість дає ізоляцію відмов (Analytics може лежати — Core продовжує працювати) та можливість незалежного масштабування воркерів.

Виняток із eventual-правила — **Gamification XP/level-up**: користувач має побачити нову кількість XP вже у відповіді на запит, що тригернув подію. Тому ці підписки явно зареєстровані з `mode="sync"` і виконуються в межах того ж циклу запиту. Аудит та проекції — `mode="async"`, бо час очікування користувача не повинен залежати від швидкості диску Analytics.

### 1.4. ADR — межі контекстів

```
# ADR-005: Modular Monolith з двома макроконтекстами (Core / Analytics)
Status: Accepted (Lab 5)
Context: HealthPatch виріс з DDD-моноліту з спільним `src/models/` та
  прямими крос-модульними імпортами. Це блокувало (а) незалежну еволюцію
  команд по доменах, (б) можливість винести Analytics на окремий процес
  у майбутньому, (в) перевірку архітектурних правил автоматично.
Decision: Поділити src/ на src/core_context/ (5 модулів) і
  src/analytics_context/ (audit + projections); експонувати з кожного
  модуля лише пакет contracts/; усі крос-модульні комунікації — через
  IntegrationEvent на спільній шині (sync для UX-критичних реакцій,
  async через arq для всього іншого); межі перевіряти import-linter в CI.
Consequences:
  + Чітка власність коду; CI стереже кордони.
  + Analytics можна винести у власний deployment без переписування Core.
  + Async-шлях природно даtime DLQ (arq failed jobs) та retry-семантику.
  − Зросла кількість файлів (contracts/, acl/, integration_publishers).
  − Eventual consistency додає затримку та ускладнює локальний дебаг.
  − Серіалізація подій у JSON для arq обмежує payload-типи.
Alternatives rejected:
  • Microservices з окремими БД — для команди з 1–2 розробників на модуль
    операційний overhead не виправданий; обираємо «monolith we can later split».
  • Залишити як було (DDD-моноліт зі спільними models) — не масштабується
    організаційно та не дозволяє автоматично перевіряти межі.
```

## 2. Ретроспектива курсу: Lab 1 → Lab 5

### 2.1. Архітектурна еволюція HealthPatch

| Лаба | Стан архітектури | Ключове досягнення |
|---|---|---|
| **Lab 1** | Класичний layered FastAPI: `routes / services / repositories / models`. Один пакет `src/models/`. Жодних доменних подій. | Базовий ВИКОНАВЧИЙ скелет: міграції Alembic, JWT, Pydantic-схеми. |
| **Lab 2** | Інтегровано Domain-Driven Design: кожен модуль отримав підкаталоги `domain / application / infrastructure / presentation`; з’явилися агрегати (`UserDomain`, `WorkoutSessionDomain`), Unit of Work. | Чітке відокремлення бізнес-логіки від HTTP та ORM. |
| **Lab 3** | CQRS-подібний поділ: окремі `commands.py` / `queries.py` / `read_models.py`. Repositories спеціалізувалися (write/read). | Read-моделі дешеві в розширенні, write-агрегати залишилися чистими. |
| **Lab 4** | Доменні події + EventBus (in-memory + arq), `dispatch_domain_events` у UoW. Винесено перші асинхронні side-effects (e-mail). | Перехід від «викликаю функцію» до «публікую подію». |
| **Lab 5** | Модульний моноліт з контрактами та ACL; макроконтекст Analytics; інтеграційні події; перевірка меж `import-linter`; гібридна шина sync/async. | Архітектурні межі стали **виконуваною** частиною коду. |

### 2.2. Найцінніші архітектурні рішення

1. **EventBus + UoW outbox (Lab 4-5).** Це єдина зміна, яка дала найбільший вихлоп: всі подальші вимоги (асинхронний аудит, проекції, нарахування XP, e-mail, лідерборд) лягли на готовий шаблон без ad-hoc коду.
2. **DDD-розшарування з самого початку (Lab 2).** Без `domain/application/infrastructure/presentation` Lab 5 був би нездійсненним: межі модулів довелось би вигадувати з нуля. Завдяки шаруванню Lab 5 звівся до «вирізати приватні залежності та підставити contracts/».
3. **Pydantic v2 frozen DTO як контракт.** Дешево в синтаксисі, безкоштовно дає валідацію, серіалізацію для arq та незмінність — ідеальний матеріал для cross-context payload.
4. **Per-handler delivery mode (`sync`/`async`) на одній шині.** Замість підтримки двох окремих імплементацій шини — один клас із параметром підписки. Знизило когнітивне навантаження і прибрало дубльовану інфраструктуру.

### 2.3. Що зробив би інакше

- **Запровадив би contracts/ ще на Lab 2.** Тоді не було б 121 крос-модульного імпорту, які довелось рефакторити в Lab 5.
- **Не тримав би `src/models/` як god-package.** Спільний пакет ORM-моделей став центром тяжіння всіх модулів і викривив межі. Краще було від початку класти ORM у `<module>/infrastructure/orm.py` і налаштовувати Alembic на discovery.
- **Виніс би аудит у власний модуль одразу.** Замість того, щоб дублювати `IAuditService` у трьох доменах і потім зливати в Analytics, відразу мав би одного централізованого споживача подій.
- **Жорсткіше відразу типізував би ID.** Половина модулів використовує `int` (Postgres serial), і в Lab 5 я ледь не пройшовся UUID-ами по всіх контрактах. Узгоджена типізація на ранній стадії — нульова робота, на пізній — рефакторинг 20 файлів.

### 2.4. Спостережені trade-offs «простота vs гнучкість»

| Питання | Простіше | Гнучкіше | Що обрав і чому |
|---|---|---|---|
| Спільні ORM моделі | Один `src/models/` | Per-module ORM | Lab 1–4: спільне (швидко). Lab 5: розпочато перенесення (готує винос модуля у власний сервіс). |
| Виклик між модулями | Прямий import + `await service.method()` | Інтеграційна подія + ACL | До Lab 4 — прямі виклики. З Lab 5 — події. Latency зросла на ~5–10 мс, але ізоляція відмов і автономність команд переважили. |
| Доставка хендлерів | Усі sync | Усі async | Гібридно: UX-критичні sync, фонові async. Уникнули як «довгих запитів від користувача», так і «де моя XP — а вона вже в черзі». |
| Перевірка меж | Code review | `import-linter` у CI | Code review пропускає 1 з 5 крос-модульних імпортів — досвід Lab 2-4. Linter ловить 5 з 5 і не втомлюється. |
| Eventually consistent read (audit/projection) | Запит у БД Core напряму | Окрема таблиця Analytics, оновлювана подіями | Окрема таблиця: дорожче в storage, дешевше в latency, і цілком повторює реальність продакшн-аналітики (data warehouse). |

### 2.5. Загальний висновок

Найважливіший урок п’яти лабораторних — **архітектурні рішення мають бути виконуваними, а не лише задокументованими**. Діаграма шарів у README не зупиняє розробника від `from src.core_context.user.domain import …`; `import-linter` зупиняє. Документ про «events for cross-module communication» не перетворює виклик функції на подію; гібридний `EventBus` із параметром `mode` робить цей перехід однією зміною підпису.

HealthPatch у фіналі курсу — не «monolith we wish we could split», а **«monolith we can split if we ever need to»**: межі формалізовано в коді, всі крос-модульні комунікації йдуть через серіалізовані події, і виокремлення Analytics в окремий процес — це питання `docker-compose.yml`, а не рефакторингу.
